# VISTA — Machine Learning Design

> **Last Updated:** 2026-06-24
> **Status:** Complete — implemented in `vista/ml/`

---

## 1. Overview

The ML module answers one question per student:
> "How likely is this student to need teacher intervention in the next 2–4 weeks?"

Two inference paths exist:
- **Rule-based** (`calculate_risk_from_metrics`) — works with zero training data
- **XGBoost** (`run_pipeline`) — trained on synthetic data, ready for real data

Both produce the same output schema. The backend can use either.

---

## 2. Dataset Requirements

### Current (Synthetic)
- 200 students × 6 columns
- Generated by `data/generate_sample_data.py` with seed=42
- Distribution: ~35% LOW, ~40% MEDIUM, ~25% HIGH
- Features: overall_attendance, recent_attendance, avg_score, recent_score, failed_subjects
- Labels derived from rule-based thresholds (self-consistent)

### Phase 2 (Real Data)
- Target: 200–400 student-semester records with actual pass/fail outcomes
- Source: UCI Student Performance Dataset (recommended in DATASET_ANALYSIS.md)
- Validation: OULAD, Kaggle Dropout dataset

## 3. Feature List (8 Engineered Features)

| # | Feature | Formula | Range |
|---|---|---|---|
| 1 | attendance_percentage | attended/total × 100 | 0–100 |
| 2 | attendance_drop_percentage | first_half_rate − second_half_rate | −100 to 100 |
| 3 | internal_marks_average | mean(score_i/max_i × 100) | 0–100 |
| 4 | marks_decline_percentage | (peak−latest)/peak × 100 | 0–100 |
| 5 | assignment_completion_rate | submitted/total × 100 | 0–100 or None |
| 6 | consecutive_absence_count | longest streak | 0–∞ |
| 7 | performance_trend | direction of last 3 scores | IMPROVING/STABLE/DECLINING |
| 8 | engagement_score | 0.6×attendance + 0.4×completion | 0–100 |

---

## 4. Feature Engineering Strategy

All features computed in `features.py` via `compute_features(StudentMetrics) → FeatureVector`.

Key design choices:
- **Normalisation:** All scores normalised to 0–100 regardless of max_score
- **Missing data:** Assignment fields nullable; engagement uses attendance-only mode
- **Trend detection:** ±5 point threshold prevents noise-driven flips
- **Override flags:** 3 boolean flags derived for critical conditions (≥7 absences, <50% attendance, <35% marks)

## 5. Model Selection

### Rule-Based Engine (Primary — Pilot)

```
risk_score = (
    (100 - attendance%) × 0.30 +
    (100 - marks_avg)   × 0.25 +
    (att_drop × 3)      × 0.15 +
    (marks_decline × 2.5) × 0.15 +
    (consec_abs × 12)   × 0.10 +
    (100 - engagement)  × 0.05
)
```

Score → Level: 0–39=LOW, 40–69=MEDIUM, 70–100=HIGH

### XGBoost (Secondary — Demo)

- 100 estimators, max_depth=4, learning_rate=0.1
- Trained on 5 features from synthetic CSV
- Macro F1 = 0.957 (test set)
- Saved as `model.pkl`

### Logistic Regression (Baseline)

- StandardScaler + L2 regularization, class_weight="balanced"
- Macro F1 = 0.94
- Kept for comparison

## 6. Training Pipeline

```python
# vista/ml/train.py
1. Load student_data.csv (200 rows × 5 features + 1 label)
2. Stratified train/test split (80/20, random_state=42)
3. Train Logistic Regression (scaled) + XGBoost (raw)
4. Evaluate both on macro F1
5. Select winner → save to model.pkl
6. Print classification report
```

Run: `python -m vista.ml.train`

## 8. Explainability Strategy

### Current (Template-Based)

Reasons generated from feature thresholds:
```python
if attendance_percentage < 75:
    reasons.append(f"Attendance at {pct:.0f}% — below 75% threshold")
```
- Max 4 reasons, ordered by weighted contribution
- Max 3 recommendations, matched to risk level

### Phase 2 (SHAP)

```python
import shap
explainer = shap.TreeExplainer(xgb_model)
shap_values = explainer.shap_values(feature_vector)
# Convert top SHAP contributors → human-readable reasons
```

Benefits:
- Per-prediction explanations (not just global importance)
- Quantifies HOW MUCH each feature contributed
- Direction-aware (pushed toward HIGH vs pushed toward LOW)

---

## 9. Override Rules (Post-Score)

Applied after scoring; can only UPGRADE level, never downgrade:

| # | Condition | Action |
|---|---|---|
| 1 | consecutive_absences ≥ 7 | Force HIGH |
| 2 | attendance < 50% | Force HIGH |
| 3 | marks_average < 35% | Force HIGH |
| 4 | attendance_drop > 25% AND level=LOW | Force MEDIUM |
| 5 | marks_decline > 30% AND level=LOW | Force MEDIUM |

## 10. ML Upgrade Path

| Phase | Model | Data Needed | Benefit |
|---|---|---|---|
| Pilot (now) | Rule-based | None | Works immediately |
| Phase 2 | Logistic Regression | 200+ real outcomes | Interpretable coefficients |
| Phase 3 | Random Forest | 500+ records | Non-linear interactions |
| Phase 4 | XGBoost + SHAP | 1000+ records | Best accuracy + per-prediction explanations |

**Design invariant:** `calculate_risk(student_id) → dict` signature NEVER changes.
Internals swap; contract stays fixed.
